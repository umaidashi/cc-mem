# FAQ

cc-mem に関するよくある質問と回答です。

---

## 基本的な質問

### Q: CLAUDE.md と cc-mem の違いは？

**CLAUDE.md** はプロジェクトのルールや指示を書く静的なファイルです。内容は手動で管理し、セッション開始時に毎回読み込まれます。

**cc-mem** は会話の履歴を自動保存し、必要なときに検索で呼び出す動的な長期記憶です。

| | CLAUDE.md | cc-mem |
|---|---|---|
| 目的 | プロジェクトルール・指示 | 過去の会話の記憶 |
| 更新方法 | 手動 | 自動 (Hook) |
| 参照方法 | セッション開始時に自動読込 | `cc-mem search` で明示的に検索 |
| データ量 | 小〜中 (1ファイル) | 大 (全セッション蓄積) |
| 用途の例 | コーディング規約、使用技術 | 設計判断の経緯、過去の議論 |

両者は補完関係にあります。CLAUDE.md に「過去の議論は cc-mem で検索せよ」と書くことで連携します。

### Q: データはどこに保存される？

```
~/.cc-mem/memory.db
```

ホームディレクトリ直下の `.cc-mem` ディレクトリに SQLite データベースとして保存されます。プロジェクトごとに分かれず、全プロジェクトの会話が1つの DB に集約されます。

### Q: クラウドにデータは送信される？

**されません。** cc-mem は完全にローカル完結です。

- データベース: ローカルの SQLite ファイル
- embedding: ローカルで動作する Ollama
- ネットワーク通信: Ollama API (`localhost`) のみ

外部サーバーへのデータ送信は一切ありません。

### Q: 短い会話も保存される？

Q と A の合計文字数が **100文字未満** の会話は自動的に除外されます。短すぎる会話は有用な情報を含まない可能性が高いためです。閾値は環境変数 `CC_MEM_MIN_CHUNK_LENGTH` で変更できます。

```bash
# 閾値を50文字に下げる
export CC_MEM_MIN_CHUNK_LENGTH=50
```

### Q: 「ありがとう」「OK」のようなやり取りは？

Q と A の **両方** が定型パターン（「ありがとう」「了解しました」等）に一致する場合は自動的に除外されます。ただし、Q が「続けて」のような軽量な発言でも、A が充実した内容であれば保存されます。フィルタは「両方が定型パターン」の場合のみ除外するため、有用な情報が失われることはありません。

### Q: 同じ質問を何度もすると重複する？

コサイン類似度が **0.95 以上** の既存メモリがある場合、自動的に保存をスキップします。完全に同じ、またはほぼ同じ内容は重複保存されません。閾値は環境変数 `CC_MEM_DEDUP_THRESHOLD` で変更できます。

```bash
# 閾値を上げて、より多くのチャンクを保存する
export CC_MEM_DEDUP_THRESHOLD=0.98
```

## 容量・パフォーマンス

### Q: DB の容量はどのくらいになる？

目安として:

| 使用量 | 会話量の目安 |
|---|---|
| 1 MB | 数十セッション |
| 10 MB | 数百セッション |
| 100 MB | 1000+ セッション |

embedding ベクトル (1チャンクあたり約3KB) が容量の大部分を占めます。`cc-mem stats` で現在の DB サイズを確認できます。

```bash
cc-mem stats
```

### Q: 古いデータを削除できる？

`cc-mem gc` コマンドで削除できます。

```bash
# デフォルト: 90日以上前のメモリを削除
cc-mem gc

# 60日以上前を削除
cc-mem gc --older-than 60d

# 特定セッションを削除
cc-mem gc --session sess_20260320_143010_1234

# 削除件数を事前確認（実際には削除しない）
cc-mem gc --dry-run
```

完全にリセットしたい場合は DB ファイルを削除してください:

```bash
rm ~/.cc-mem/memory.db
rm -f ~/.cc-mem/memory.db-wal
rm -f ~/.cc-mem/memory.db-shm
```

### Q: 検索が遅くなることはある？

ベクトル検索は全ての embedding に対してコサイン類似度を計算するため、データ量に比例して遅くなります。ただし、数千件程度であれば体感できるほどの遅延にはなりません。

FTS5 検索は trigram インデックスを使用しているため、データ量が増えても高速です。

## Ollama 関連

### Q: Ollama なしでも使える？

**使えます。** Ollama が未起動の場合:

- **保存:** embedding なし (`null`) で保存される
- **検索:** FTS5 全文検索（trigram）のみが機能する

ベクトル検索（意味的類似度）は使えませんが、キーワードベースの検索は問題なく動作します。

### Q: nomic-embed-text 以外のモデルは使える？

環境変数 `CC_MEM_EMBED_MODEL` で変更できます:

```bash
export CC_MEM_EMBED_MODEL="mxbai-embed-large"
ollama pull mxbai-embed-large
```

ただし、モデルを変更すると既存のベクトルとの次元数が異なる場合があり、検索精度が低下する可能性があります。モデルを変更する場合は DB をリセットすることを推奨します。

### Q: Ollama を常時起動しておく必要がある？

保存時と検索時に Ollama が起動していれば十分です。macOS で Ollama.app をインストールしている場合は、ログイン時に自動起動する設定が可能です。

## Hook 関連

### Q: 全てのセッションが保存される？

Stop Hook は Claude Code がレスポンス生成を完了するたびに発火します。つまり、1つのセッション中で複数回 save が実行される可能性があります。

同一セッション内で同じ会話が複数回パイプされますが、重複排除機能（コサイン類似度 0.95 以上で自動スキップ）により、同じ内容が重複保存されることはありません。

### Q: 特定のプロジェクトだけ保存したい場合は？

`matcher` フィールドに正規表現を設定することで、特定の条件でのみ Hook を発火させることができます。ただし、`matcher` はレスポンス内容に対するフィルタであり、プロジェクトパスによるフィルタは現時点ではサポートされていません。

全プロジェクトで保存し、検索時にキーワードで絞り込むのが推奨の運用方法です。

### Q: Hook の実行でClaude Codeが遅くなる？

コマンド末尾の `&` によりバックグラウンドで実行されるため、Claude Code のレスポンスには影響しません。`2>/dev/null` により stderr も抑制されます。

## 検索関連

### Q: 検索スコアの意味は？

検索結果に表示される 0-100% のスコアは、以下の要素を統合した RRF スコアを正規化した値です:

1. **FTS5 ランク** -- キーワードの一致度（BM25）
2. **ベクトル類似度** -- 意味的な類似度（コサイン類似度）
3. **RRF 統合** -- 上記2つのランクを Reciprocal Rank Fusion で統合
4. **時間減衰** -- 新しい記憶ほどスコアが高い（半減期30日）

最もスコアの高い結果が 100% に近い値になるよう正規化されます。相対的な関連度の目安として参照してください。

### Q: 英語の会話も検索できる？

はい。FTS5 trigram トークナイザは言語に依存しないため、日本語・英語・その他の言語が混在していても検索可能です。ベクトル検索も `nomic-embed-text` が多言語に対応しています。

## プロジェクトスコープ関連

### Q: プロジェクトスコープとは？

search と recall はデフォルトで現在のプロジェクトにスコープされます。プロジェクト名は CWD（現在のディレクトリ）の basename から自動取得されます。例えば `/Users/alice/projects/my-app` で実行すると、プロジェクト名は `my-app` になります。

### Q: 全プロジェクトの記憶を検索したい場合は？

`--all` フラグを付けてください。

```bash
cc-mem search --all "設計判断"
cc-mem recall --all
```

### Q: プロジェクト名を変更したい場合は？

環境変数 `CC_MEM_PROJECT` で上書きできます。monorepo のように CWD の basename が意図したプロジェクト名にならない場合に便利です。

```bash
export CC_MEM_PROJECT="my-service"
```

### Q: スコープ導入前のデータはどうなる？

プロジェクトスコープ導入前に保存されたデータ（project が空文字のもの）は、どのプロジェクトからでも検索対象に含まれます。データの移行作業は不要です。

## Import 関連

### Q: import は何をするの？

`cc-mem import` は `~/.claude/projects/` 配下に蓄積された Claude Code のセッションログ（.jsonl ファイル）を一括で cc-mem に取り込みます。cc-mem 導入前の過去の会話も記憶として活用できるようになります。

### Q: import を何度も実行しても大丈夫？

はい。重複排除機能により冪等です。同じセッションログを複数回取り込んでも、既に保存済みの記憶は自動的にスキップされます。

### Q: import 前に件数を確認したい場合は？

`--dry-run` フラグを使ってください。実際には保存せず、処理対象のセッション数のみ表示します。

```bash
cc-mem import --dry-run
```

### Q: 特定のプロジェクトだけ import したい場合は？

`--project` フラグでプロジェクト名を指定できます。

```bash
cc-mem import --project my-app
```
