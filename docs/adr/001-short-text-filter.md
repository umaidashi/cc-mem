# ADR-001: 短文フィルタリング

## Status
Accepted

## Context
cc-mem は会話終了時に Q&A チャンクを SQLite に保存するが、現状の chunker は Q と A が空でなければ全て保存する。「ファイル読んで」「はい」「OK」のような短い Q&A はノイズとなり、検索精度を下げる原因になっている。短い Q&A は文脈的に意味のある情報を含まないため、保存前にフィルタリングする仕組みが必要である。

## Decision
- Q+A の合計文字数が閾値未満のチャンクを保存対象から除外する
- 閾値のデフォルト値は **100 文字**（日本語で約 2〜3 文に相当）
- 閾値は環境変数 `CC_MEM_MIN_CHUNK_LENGTH` で変更可能とする
- フィルタリングは chunker の後段、保存処理の直前に適用する

## Acceptance Criteria
1. Q+A 合計 100 文字未満のチャンクは保存されない
2. Q+A 合計 100 文字以上のチャンクは保存される
3. 閾値は環境変数 `CC_MEM_MIN_CHUNK_LENGTH` で変更可能
4. フィルタ前後のチャンク数が stderr に出力される（「5件中3件を保存」のような形式）
5. 全チャンクがフィルタされた場合は「保存するメモリがありません」と表示

## Test Plan
1. **閾値未満の除外テスト**: Q=「読んで」(3文字) + A=「はい」(2文字) = 合計5文字のチャンクを作成し、保存後に SQLite を検索して存在しないことを確認する
2. **閾値以上の保存テスト**: Q+A 合計 100 文字以上のチャンクを作成し、保存後に SQLite を検索して存在することを確認する
3. **境界値テスト**: Q+A 合計がちょうど 99 文字（除外）と 100 文字（保存）のチャンクで動作を確認する
4. **環境変数テスト**: `CC_MEM_MIN_CHUNK_LENGTH=50` を設定し、合計 50 文字以上で保存・50 文字未満で除外されることを確認する
5. **stderr 出力テスト**: 5件のチャンク（うち2件が閾値未満）を処理し、stderr に「5件中3件を保存」相当のメッセージが出力されることを確認する
6. **全除外テスト**: 全チャンクが閾値未満の場合に「保存するメモリがありません」が stderr に出力されることを確認する

## Consequences

### メリット
- ノイズとなる短文 Q&A が排除され、検索精度が向上する
- SQLite のデータ量が削減され、ストレージ効率が改善する
- 環境変数で閾値を調整できるため、ユーザーごとの運用に対応可能

### デメリット
- 短いが有用な Q&A（例: Q=「exit code 1の意味は？」A=「異常終了」）がフィルタされる可能性がある
- 閾値のデフォルト値（100文字）が全てのユースケースに最適とは限らない
