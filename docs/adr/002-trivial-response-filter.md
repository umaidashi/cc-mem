# ADR-002: 軽量応答の除外

## Status
Accepted

## Context
「はい」「OK」「ありがとう」「了解」のような定型的な質問・応答は記憶する価値がない。質問側も「続けて」「お願い」のような短い指示は情報量が低い。これらが蓄積すると検索結果の質が低下する。ADR-001 の短文フィルタでは捕捉できないケース（例: A が定型的な長文応答「どういたしまして！他に質問があればどうぞ。」）にも対応する必要がある。

## Decision
- 定義済みの軽量パターンリスト（正規表現）に一致する Q/A を検出する
- **Q と A の両方が軽量パターンに一致する場合のみ除外する**
- Q が軽量でも A が充実していれば保存する（A 側だけで除外判定しない）
- 軽量パターンの例:
  - Q 側: `^(はい|OK|ok|おk|続けて|お願い(します)?|いいね|よろしく|ありがとう(ございます)?|了解|承知(しました)?|うん|ええ|頼む|やって|それで)[\s。！!]*$`
  - A 側: `^(はい|OK|了解(しました|です)?|承知(しました|です)?|わかりました|かしこまりました|どういたしまして[！!。]?.*|他に(質問|ご質問)があれば.*|お役に立てて.*|ご不明な点があれば.*)[\s。！!]*$`
- パターンリストは設定ファイルまたはコード内の定数として管理し、拡張可能な設計とする

## Acceptance Criteria
1. Q=「続けて」A=「了解しました。以下に詳細を説明します...（長い回答）」 → 保存される（A が充実しており軽量パターンに一致しない）
2. Q=「OK」A=「はい」 → 除外される（Q・A ともに軽量パターンに一致）
3. Q=「ありがとう」A=「どういたしまして！他に質問があればどうぞ。」 → 除外される（Q・A ともに軽量パターンに一致）
4. Q=「ReactのuseStateとは？」A=「useStateは...（長い説明）」 → 保存される（Q が軽量パターンに一致しない）
5. パターンリストは拡張可能な設計である（新しいパターンの追加がコード1箇所の変更で完結する）

## Test Plan
1. **A 充実テスト**: Q=「続けて」、A=200文字以上の技術的回答を持つチャンクを処理し、保存されることを確認する
2. **両方軽量テスト**: Q=「OK」、A=「はい」のチャンクを処理し、除外されることを確認する
3. **定型応答テスト**: Q=「ありがとう」、A=「どういたしまして！他に質問があればどうぞ。」のチャンクを処理し、除外されることを確認する
4. **通常 Q&A テスト**: Q=「ReactのuseStateとは？」、A=「useStateは...」の技術的 Q&A が保存されることを確認する
5. **パターン拡張テスト**: 新しいパターン（例: 「サンキュー」）をリストに追加し、追加後に該当チャンクが除外されることを確認する
6. **ADR-001 との組み合わせテスト**: 短文フィルタと軽量応答フィルタの両方を有効にし、各フィルタが独立して機能することを確認する

## Consequences

### メリット
- 定型的なやり取りが排除され、検索結果の質が向上する
- ADR-001 の短文フィルタでは捕捉できない「長いが無意味な定型応答」も除外できる
- Q が軽量でも A が充実していれば保存するため、有用な情報の損失を防げる

### デメリット
- パターンリストのメンテナンスが必要（新しい定型表現への対応）
- 日本語以外の言語への対応が別途必要になる
- 正規表現の設計によっては、意図しないチャンクが除外される可能性がある
